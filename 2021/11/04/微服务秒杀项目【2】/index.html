3. 完善网关

  • 通过前端login.htm登录页面定义的ajax可知，访问路径是member/token/login
    • member：从码云获取的路由路径，实际上访问的就是member-server
    • token：代表令牌的意思，需要我们定义
    • login：登录操作
token/login定义

  • 在member-server/Controller中新建UserController类（/token映射）

    • 添加@RestController注解

    • 创建/login映射的login方法

      • 登录之后我们需要获取用户的全部信息，因为后面需要参与秒杀，还需要对订单进行操作
        • 所以我们login方法的返回值不能使用String，应该具有
          • 状态码
          • 登录状态信息
          • 数据（id,name等等）
        • 所以我们需要自定义一个类来保存用户登录的信息（登录成功、失败信息类）
          • 这些信息在用户服务，秒杀服务，商品服务都都有可能使用
          • 所以我们需要把它定义在common公共微服务中（具体操作请看目录）

    • 在一系列的错误异常信息封装、处理等操作后，最后返回的时一个Result类型的数据，所以我们将login方法的返回值更改为Result< String>

    • 通过前端页面可知，前端传递了两个参数分别是username和password

      • 但页面的Id分别是mobile和password，并且为了确保数据的安全性，所以直接使用username并不是最佳方案，我们需要在member-server中自定义一个Web/Controller/LoginVo类（下面，找不到看目录）

    • 更改login登录方法的参数列表，添加一个LoginVo对象

    • 注入一个IUserService的对象，用于注册时调用方法

    • 反向在IUserService中生成login登录方法

    • UserService中重写接口中新添加的login方法

      • 通过调用本类的find方法，查询数据库是否有当前输入的手机号

      • 判断find获取到的user是否为空，如果为空

        • 如果为空，我们就需要创建一个异常信息并返回，所以我们需要新建一个MemberServerContreoolerAdvice异常类进行操作（之前定义的BusinessException是一个异常的基类，相当于Exception和IOException的关系）（新建异常类操作在下面，找不到看目录）
        • 抛出新建的异常

    数据库中slat盐的作用：密码在进行加密算法时，通常常用的12345678等的密文都一样，如果使用穷举的方式，很容易就会破解，所以需要使用密码+salt盐 的方式

    • 为了解决密码加密的问题，我们需要在member-server/web包下新建md5s/MD5Util类，进行操作（下面，找不到看目录）

    • 还需要对提交密码是否为空进行判断

      • 使用JSR303进行校验

        • 主要对JavaBean参数进行校验，即能校验是否为空，还可以校验参数是否合法

      • 导入依赖

      • 在LoginVo（用户提交数据的封装类）中添加注解进行使用

        • @NotNull 必须时非空的
        • @Null 必须是空
        • @Digits 验证数字的构成是否合法
        • @Future 验证是否在系统当前时间之后
        • @Past 验证是否在系统当前时间之前
        • @Max 是否比当前数值大
        • @Min 是否比当前数值小
        • @Pattern 验证正则表达式
        • @Size 验证大小是否在指定的长度范围内
        • @DecimalMax 验证值是否小于等于最大值
        • @DecimalMin 验证值是否大于等于最小值

      使用JSR303的前提：JavaBean被序列化

      • 还需要在UserController的login方法参数声明上添加标签@Valid验证
      • 借助通知类进行信息展示

成功信息类定义

  • 成功信息类：在common公共包中新建Result泛型类

    • 实体类为了方便使用lombok，给公共项目添加lombok依赖

      • 再添加一个springweb启动依赖，后面要用

    • 定义状态码、消息、数据信息（类型不确定，使用泛型）等成员变量

    • 定义固定的状态码

    • 添加带参构造（没截图）

    • 还需要定义成功时的方法
      • 分别为固定的成功状态码，只需要传递data登陆数据
      • 还有一种需要传递状态码和data数据

    • 还需要定义失败时的方法

      • 因为失败时所需要的返回信息和成功不同，并且错误类型比较多，无法进行预先的全局定义，所以我们还需要重新定义一个类来封装失败信息 （具体操作请看目录）
      • 定义发生错误时的方法

      • 定义发生错误时的默认方法

      • 判断是否存在错误

失败信息类定义

  • 失败信息类：在common公共包中新建CodeMsg类

    • 添加@Getter/@Setter注解

  • 一旦Service出现问题，Controller层只负责调用和接收数据，无法判断

    • 我们需要将Service进行RuntimeException处理，并且增删改需要进行回滚操作
      • 回滚操作：需要定义一个异常类进行完善（在下面，找不到请看目录）
    • 所以需要在失败信息封装类中预定义两个RuntimeException异常错误消息

成功和失败信息类，应该在公共类中添加一个common包存放

使用postman等类似工具时，可以越过前端页面的JS、JQ等验证进行登录

但是越不过service层的业务处理，所以我们定义的RuntimeException和CodeMsg就起到二次认证的作用

异常类定义（基类）

  • 异常类：在common公共项目common包中新建BusinessException类

    • 继承RuntimeException

    • 添加@Getter/@Setter注解
异常统一处理类定义

service抛异常时会抛给控制器，如果不进行处理前端页面就会报错，所以我们还需要新建一个类对所以的控制器异常进行统一处理

  • 在common公共项目common包中新建CommonControllerAdvice类

    • 添加@ControllerAdvice
      • 这个注解相当于对controller添加了try catch
    • 分别创建error的异常信息和defaulterror（默认）的异常信息

这个类可以作为一个父级别的类，再后面的服务中可以使用到

登陆时需要的用户名和密码

VO就是用于展示的数据 Value Object

  • 这里见member-server中的controller放入了新建的web包下

  • 并在Web报下新建vo/LoginVo

    • 添加成员变量，添加Getter/Setter

用户服务异常类通知

    • 继承于CommonControllerAdvice异常统一处理类
    • 添加@ControllerAdvice注解

  • 在member-server中的web包下，新建code/MemberCodeMsg异常类

    • 继承于CodeMsg，创建构造方法
    • 新建一些异常操作方法
