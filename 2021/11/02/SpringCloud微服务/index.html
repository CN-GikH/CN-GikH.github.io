一、SpringCloud

1.概念

  • 微服务架构
  • 系统架构设计阶段分为两个阶段
    • 集中式系统
      • 集中式系统也叫单体应用，功能、模块集中到一个项目中
      • 部署到一个服务器上，对外提供服务
    • 分布式系统
      • 分布式系统就是讲功能按照模块进行拆分，形成不同的子系统（微服务）
      • 部署到不同的服务器上（地域都可能不同）
      • 这些系统需要协作（相互调用）

  • 什么是微服务？

    https://blog.csdn.net/qq_32252957/article/details/90247418

    • 简单来说，就是一种架构的设计风格
      • 将原来的独立的集中式系统，拆分成多个小型服务
      • 这些小型服务基于Http的restful api进行访问
      • 这些围绕某一个业务模块功能，并且它是一个独立的项目，可以独立的开发和部署
  • 为什么学习Spring Cloud？
    • 微服务开发的问题
      • 服务基础开发：SpringMVC、Spring、SpringBoot
      • 服务的注册与发现：
        • RPC： Dubbo 、Feign、ribbon、Eureka
      • 分布式需要管理配置： Netflix Config
      • 负载均衡： ribbon 配合 Nginx
      • 服务熔断： Hystrix （使用熔断中的降级）
      • API网关： Zuul
      • 批量任务： 当当elastic job
      • 服务跟踪： 京东服务跟踪Hydra
    • Spring Cloud 对上述所有问题，提供了一站式解决方案

  • Spring Cloud 是什么？

    https://spring.io/projects/spring-cloud

    • 一站式分布式系统开发框架，提供一系列的分布式框架工具集

    • H版和G版用的比较多
      • SpringBoot版本不能低于SpringCloud的版本
    • Spring Cloud 集成Netflix公司的各种组件

3.服务注册与发现中心 Eureka

  • SpringCloud中提供了三种服务注册中心
    • Eureka （支持最好的）
    • Consul
    • Zookeeper
  • 注册中心，注册如下信息：
    • 所在主机，端口，版本号，通信协议等都登记到注册中心
  • 服务发现
    • 服务的消费者向注册中心请求已经登记过的服务列表
      • 然后得到某个服务主机、端口号
  • Eureka是什么？
    • 服务治理的组件：提供服务注册和服务发现
      • 用它来搭建 服务注册 与 发现中心
    • 它也是基于REST的服务，定位我们的服务位置，进行中间层的服务器的负载均衡
    • Eureka是Netflix公司开发的，被SpringCloud进行二次封装
    • Eureka是 C/S 架构的
      • 有服务端 Server
      • 有客户端 Client

4.Eureka入门

  • 创建server项目

    • 添加spring web模块
    • 添加Eureka Server模块
      • 可以理解为服务器端

    • 如果没有添加Eureka Server模块，需要手动在pom.xml文件中添加

  • 在Application中添加@EnableEurekaServer注解，开启Eureka服务

  • 在application.properties中添加配置

    • 添加端口号配置，最好是91开头

    • 添加项目路径 /eureka-server

      • 注意：要使用spring.application.name

    • 设置服务注册中心

    • 设置不能让它自己向自己注册（注册和发现中心需要接收别人的注册或发现）

    • 主要作用是维护服务，不是主动检索服务

    • 公开注册中心地址

  • 创建server-provider项目（提供者）

    • 添加Spring Web模块

    • 添加Eureka Discovery Client模块

      • 可以理解为客户端

    • 配置application.properties

    • 添加Spring Web、Eureka Discovery Client模块

5.CAP理论

CAP是什么
  • C：一致性
  • A：可用性
  • P：容错性（分区容错）

在分布式项目中，CAP不可能同时存在

  • CA、CP、AP
    • 又因为A是可用性，必须要有
    • 分区容错（P）是分布式项目必须要保证的
      • 因此只能A、C之间选
  • Zookeeper和Eureka的区别
    • Zookeeper是CP
      • 当master节点因为网络故障与其他的节点失联，剩下的节点就会重新”选举”，但是有问题
        • 需要时间：不为外界提供服务，整个Zookeeper是不可用状态
          • 导致注册的服务全部瘫痪
    • Eureka是AP
      • 优先保证可用，各个节点都是平等的
      • 某个节点或者某几个节点挂掉了不影响其他节点的工作
